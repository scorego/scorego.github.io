一、概述

MongoDB是一种面向文档的数据库管理系统，用C++等语言撰写而成，旨在为WEB应用提供可扩展的高性能数据存储解决方案。MongoDB名字来自于英文单词”Humongous”，中文含义为“庞大”。MongoDB的数据模式可以随着应用程序的发展而灵活地更新，与此同时，它也为开发人员提供了传统数据库的功能：二级索引、完整的查询系统以及严格一致性等等。MongoDB能够使企业更加具有敏捷性和可扩展性，各种规模的企业都可以通过使用MongoDB来创建新的应用，提高与客户之间的工作效率，加快产品上市时间，以及降低企业成本。
MongoDB是专为可扩展性，高性能和高可用性而设计的数据库。它可以从单服务器部署扩展到大型、复杂的多数据中心架构。利用内存计算的优势，MongoDB能够提供高性能的数据读写操作。 MongoDB的本地复制和自动故障转移功能使应用程序具有企业级的可靠性和操作灵活性。
MongoDB使用内存映射文件, 32位系统上限制大小为2GB的数据，64位支持更大的数据。MongoDB的核心构件只能在小端序系统上运行，客户端库可以在小端序和大端序的系统上运行。

Document Database：
MongoDB中的记录是一个文档，它是一个由字段和值对组成的数据结构。MongoDB文档类似于JSON对象。字段的值可以包括其他文档、数组和文档数组。

使用文档的优点是:

  • 文档(即对象)对应于许多编程语言中的本地数据类型。
  • 嵌入式文档和数组减少了对损耗性能的JOIN的需求。
  • 动态模式支持连贯多态性。

Key Features：

  • High Performance
    MongoDB提供高性能的数据持久性。特别是，

    • 对嵌入式数据模型的支持减少了数据库系统上的I/O活动。
    • 索引支持更快的查询，并且可以包含来自嵌入式文档和数组的键。

  • Rich Query Language
    MongoDB支持丰富的查询语言，支持读和写操作(CRUD)，以及:

    • Data Aggregation
    • Text Search and Geospatial Queries.

  • High Availability
    MongoDB的replication facility，称为replica set(副本集)。副本集是一组MongoDB服务器，它们维护相同的数据集，提供冗余并提高数据可用性。它们提供了以下功能：

    • automatic failover and
    • data redundancy.

  • Horizontal Scalability
    MongoDB的水平可扩展是其核心功能的一部分:

    • 分片(Sharding)将数据分布在多个机器集群中。
    • 从3.4开始，MongoDB支持基于 shard key键创建zone。在一个平衡的集群中，MongoDB将一个区域所覆盖的读写只定向到该区域内的那些碎片。

  • Support for Multiple Storage Engines
    MongoDB支持多个存储引擎。此外，MongoDB提供了可插入存储引擎API，允许第三方为MongoDB开发存储引擎。

    • WiredTiger Storage Engine (including support for Encryption at Rest)
    • In-Memory Storage Engine
    • MMAPv1 Storage Engine (Deprecated in MongoDB 4.0)

2.1 database

一个MongoDB可以创建多个数据库，默认数据库为db，存在data目录中。MongoDB的单个实例可以容纳多个独立的数据库，每一个都有自己的集合和权限，不同的数据库也放在不同的文件中。数据库通过名字来标识。

需要注意的是：

2.3 collection

集合就是文档组，类似于RDBMS中的table。

  • capped collection
    Capped collection就是固定大小的collection，它可以支持对文档按插入顺序来进行插入和查找的高吞吐量操作，capped collection的工作方式类似于循环的buffer:一旦集合填充了分配的空间，它就通过覆盖集合中最老的文档为新文档腾出空间。
    Capped collections 高性能地自动维护对象的插入顺序，非常适合类似记录日志的功能。和标准的collection不同，必须要显式的创建一个capped collection，指定一个collection的大小，单位是字节。collection 的数据存储空间值提前分配的。
    Capped collections可以按照文档的插入顺序保存到集合中，而且这些文档在磁盘上存放位置也是按照插入顺序来保存的，所以当我们更新Capped collections中文档的时候，更新后的文档与之前文档的大小不同时，更新会失败。
    如果要在capped collection上进行update操作，那么应该创建索引以避免遍历。
    要注意的是指定的存储大小包含了数据库的头信息。

3.1 ObjectId

ObjectId很小、可能惟一、生成快，并且是有序的。ObjectId值由12个字节组成，其中前四个字节是反映ObjectId创建的时间戳。具体地：

  • 一个4字节的值表示自Unix历元以来的秒数
  • 一个5字节的随机值
  • 一个3字节的计数器，从一个随机值开始。

在MongoDB中，存储在集合中的每个文档都需要一个惟一的_id字段作为主键。如果插入的文档省略了_id字段，MongoDB驱动程序将自动为_id字段生成ObjectId。

3.2 String

BSON string编码是UTF-8。通常，在序列化和反序列化BSON时，每种编程语言的驱动程序都会从该语言的字符串格式转换为UTF-8。这使得在BSON字符串中轻松存储大多数国际字符。此外，MongoDB $regex查询支持UTF-8编码的regex string。

3.3 Timestamp

BSON有一个MongoDB内部使用的特殊时间戳类型，与常规的日期类型无关。在单个mongod实例中，Timestamp值总是惟一的。Timestamp值是一个64 bit的值，其中:前32位是time_t值(从Unix纪元开始的秒),后32位是给定秒内操作的递增序号。如果在顶级字段中插入一个包含空BSON时间戳的文档，MongoDB服务器将用当前时间戳值替换该空时间戳。如果是嵌入文档中的字段，服务器会将其保留为空的时间戳值。

3.4 Date

BSON Date是一个64位整数，表示自Unix纪元(1970年1月1日)以来的毫秒数。这支持一个大约2.9亿年的日期范围。BSON Date是有符号数，负数表示1970年之前的时间。

C. Array

四、 MongoDB Extended JSON

JSON类型是BSON支持的类型的子集。为了保存类型信息，MongoDB在JSON格式中添加了以下扩展:

  • 严格的模式。BSON类型的严格模式符合JSON RFC。任何JSON解析器都可以将这些严格模式的表示解析为键值对;但是，只有MongoDB内部的JSON解析器能够识别格式所传递的类型信息。
  • mongo Shell模式。MongoDB内部JSON解析器和mongo shell可以解析这种模式。

4.1 Parsers and Supported Format

A. Input in Strict Mode

下述可以解析严格模式的表示，并识别类型信息：

  • REST Interfaces
  • mongoimport
  • –query option of various MongoDB tools
  • MongoDB Compass

其他JSON解析器，包括mongo shell和db.eval()，可以将严格的模式表示解析为键值对，但不能识别类型信息。

4.2 BSON Data Types and Associated Representations

下面介绍了严格模式和mongo Shell模式下的BSON数据类型和相关表示。

A. Binary

data_binary：

<bindata>是二进制字符串的base64表示形式。
<t>是表示数据类型的单个字节。在严格模式下，它是一个十六进制字符串，在Shell模式下，它是一个整数。

B. Date

data_date:

在严格模式下，<date>是ISO-8601日期格式，强制时区字段遵循模板YYYY-MM-DDTHH:mm:ss.mmm<+/-Offset>。
在Shell模式中，<date>是一个64位带符号整数的JSON表示，给出了从UTC历元开始的毫秒数。

C. Timestamp

data_timestamp：

<t>是一个32位无符号整数的JSON表示形式，从历元开始，以秒为单位。
<i>是一个32位无符号整数。

D. Regular Expression

data_regex:

<sRegex>是一个由有效JSON字符组成的字符串。
<jRegex>是一个字符串，可能包含有效的JSON字符和未转义的双引号(“)字符，但可能不包含未转义的正斜杠(/)字符。
<sOptions>是一个字符串，包含用字母表示的正则表达式选项。
<jOptions>是一个字符串，可能只包含字符’g’、’i’、’m’和’s’。由于JavaScript和mongo Shell表示支持有限范围的选项，因此在转换为这种表示时，任何不符合约定的选项都将被删除。

E. OID

data_oid:

<id>是一个24个字符的十六进制字符串。

F. DB Reference

data_ref：

<name>是一个由有效JSON字符组成的字符串。
<id>是任何有效的扩展JSON类型。
