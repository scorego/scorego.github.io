Java并发05 JUC包中的锁

发表于 2019-06-05 | 更新于 2019-07-29 | 分类于 Java并发


JUC锁位于java.util.concurrent.locks包下，为锁和等待条件提供一个框架，它不同于内置同步和监视器。CountDownLatch、CyclicBarrier和Semaphore不在此包中，但也是通过AQS来实现的，因此也将它们归纳到JUC锁这一大栏中。

1. Lock接口

源码：

Lock接口的实现提供了比使用synchronized更广泛的锁操作。它允许更灵活的结构，并且可能支持多个关联的Condition对象。它支持那些语义不同(重入、公平等)的锁规则。所谓语义不同，是指锁可有”公平机制的锁”、”非公平机制的锁”、”可重入的锁”等等。
锁是控制多个线程对共享资源访问的工具，通常锁提供对共享资源的独占访问，每次只有一个线程可以获得锁，而对共享资源的所有访问都要求首先获得锁。(有些锁可能允许并发访问共享资源，比如ReadWriteLock的读锁。)
使用synchronized提供与每个对象关联的隐式监视器锁,但它强制所有锁获取与释放都是块结构的:获得的多个锁必须以相反的顺序释放，且所有的锁都必须在与获取锁的相同的词法作用域内释放。虽然synchronized的作用域机制使使用监视器锁进行编程变得容易得多，并有助于避免涉及锁的许多常见编程错误，但某些情况需要以更灵活的方式使用锁。例如，并发遍历一些数据结构的算法需要使用hand-over-hand或chain lock:先获取节点A的锁，然后是节点B，然后释放A并获取C，然后释放B并获取D，以此类推。这时可以使用Lock接口，Lock接口的实现允许在不同的范围内获取和释放锁，并允许以任何顺序获取和释放多个锁。
这种灵活性的增加带来了额外的责任。块结构的缺失导致其不会像synchronized锁那样自动释放。在大多数情况下，应该使用以下习语:

当锁定和解锁发生在不同的范围时，必须注意确保锁被持有时执行的所有代码都受到try-finally或try-catch的保护，以确保锁在必要时被释放。
Lock的实现通过提供一个非阻塞的尝试(tryLock())、试图获得一个可interrupt的锁(lockInterruptibly())，和可超时的试图获取锁(tryLock(long,TimeUnit))来提供比synchronized更多的功能。
Lock类还可以提供与隐式监视器锁截然不同的行为和语义，例如保证顺序、不可重入或死锁检测。如果一个实现提供了这种特殊的语义，那么实现必须记录这些语义。
注意，Lock的实例只是普通对象，它们本身可以用作synchronized的目标。获取Lock实例的监视器锁与调用该实例的lock()方法没有特殊的关系。为了避免混淆，建议永远不要以这种方式使用Lock的实例，除非在它们自己的实现中使用。
所有的锁Lock实现都必须执行与内置监视器锁提供的相同的内存同步语义，具体描述在The Java Language Specification (17.4 Memory Model)：

  • A successful lock operation has the same memory synchronization effects as a successful Lock action.
  • A successful unlock operation has the same memory synchronization effects as a successful Unlock action.

不成功的加锁和解锁操作以及可重入的锁定/和解锁操作不需要任何内存同步效果。

三种形式的锁获取(可中断、不可中断和超时)在性能特征、顺序保证或其他实现质量方面可能有所不同。此外，中断正在进行的获取锁的功能可能在给定的Lock类中不可用。因此，此接口的实现不需要为所有三种形式的锁获取定义完全相同的保证或语义，也不需要支持中断正在进行的锁获取。实现类需要清楚地记录每种锁定方法提供的语义和保证。如果要支持锁获取中断，它还必须遵守这个接口中定义的中断语义:要么完全支持，要么只在方法入口支持。
由于中断通常意味着取消，并且对中断的检查通常不频繁，因此实现类更倾向于响应中断而不是正常的方法返回。即使中断发生在可能已经解除了线程阻塞的另一个操作后，这也是正确的。实现类应该记录这种行为。
除非特别指出，为任何参数传递null值都会引发NullPointerException。

2. ReadWriteLock接口

源码：

ReadWriteLock接口以和Lock类似的方式定义了一些读取者可以共享而写入者独占的锁。JUC包只有一个类实现了该接口，即ReentrantReadWriteLock。
ReadWriteLock维护一对关联的锁，一个用于只读操作，另一个用于写操作。读锁可以由多个读线程同时持有，只要没有写线程。写锁是独占的。
所有ReadWriteLock的实现都必须保证writeLock操作的内存同步效果相对于关联的readLock也有效。也就是说，成功获取读锁的线程将看到在以前的写锁版本上所做的所有更新。
读写锁允许在访问共享数据时具有比互斥锁更高级别的并发性。它的依据是:虽然每次只有一个线程(写线程)可以修改共享数据，但在许多情况下，任何数量的线程都可以并发地读取数据(读取线程)。理论上，与使用互斥锁相比，使用读写锁所允许的并发性的增加将导致性能的改进。实际上，这种并发性的增加只有在多处理器系统上才能完全实现，而且只有在共享数据的访问模式合适时才能实现。
读写锁是否会相对于使用互斥锁提高性能取决于读和写的相对频率、读和写操作的持续时间以及对数据的争用——也就是说同一时刻尝试读或写数据的线程数量。例如，一个集合最初使用数据填充，之后很少修改，而且经常被检索，这是使用读写锁的理想选择。然而，如果更新变得频繁，那么数据的大部分时间都是完全锁定的，并发性几乎没有增加。此外，如果读操作太短，读写锁实现的开销(其本质上比互斥锁更复杂)可能会成为主要的开销，特别是因为许多读写锁实现仍然通过一小段代码序列化所有线程。总之，只有分析和度量才能确定读写锁的使用是否适合一个应用程序。
尽管读写锁的基本操作很简单，但是读写锁的实现必须做出许多策略决策，这可能会影响给定应用程序中读写锁的有效性。这些策略的例子包括:

  • 在读线程和写线程都在等待时，确定在释放写锁时授予读锁还是写锁。普遍会选择写线程，因为一般认为写线程耗时短且不频繁的。而偏好读线程则不那么常见，因为如果读线程如预期的那样频繁且耗时长，则会导致写作的长时间延迟。另外公平或“有序”的选择也是可以的。
  • 确定当读取器处于活动状态且写入器正在等待时，请求读锁的读取器是否被授予读锁。偏向读写器可能无限期地延迟写入器，而偏向写入器可能降低并发度。
  • 确定锁是否可重入:具有写锁的线程能否重新获得它?它能在持有写锁的同时获得读锁吗?读锁本身是可重入的吗?
  • 是否可以在不允许插入写入器时将写锁降级为读锁?相对于其他等待的读取器或写入器，读锁是否可以升级为写锁?

在实际使用读写锁时，这些因素都要考虑在内。

3. Condition接口

源码：

Condition需要和Lock联合使用，它的作用是代替Object监视器方法，可以通过await()、signal()来休眠/唤醒线程。Condition接口描述了可能会与锁有关联的条件变量。这些变量在用法上与使用Object.wait访问的隐式监视器类似，但提供了更强大的功能。需要特别指出的是，单个Lock可能与多个Condition对象关联。为了避免兼容性问题，Condition方法的名称与对应的Object版本中的不同。
Condition将Object monitor methods(wait、notify和notifyAll)分解为不同的对象，通过将它们与任意Lock实现相结合，从而实现每个对象具有多个等待集的效果。Lock替换synchronized的使用，Condition替换Object monitor methods的使用。
Condition(也称为condition queues或condition variables)提供了一种方法，让一个线程暂停执行(wait)，直到被另一个线程通知某个状态条件现在可能为true。由于对共享状态信息的访问发生在不同的线程中，因此必须对其进行保护，因此某种形式的锁与condition相关联。wait for a condition提供的关键属性是它自动释放关联的锁并挂起当前线程，就像Object.wait一样。
Condition实例本质上绑定到锁。要获取特定Lock实例的Condition实例，请使用它的newCondition()方法。

注意Condition的实现只是普通的类，也可以用作synchronized的目标。获取Condition实例的监视锁，或使用其监视方法，与获取与该Condition关联的Lock或使用其等待和发送信号的方法没有特殊的关系。为了避免混淆，建议永远不要以这种方式使用Condition实例，除非在它们自己的实现中使用。
除非特别指出，为任何参数传递null值都会引发NullPointerException。

在等待条件时，作为对底层平台语义的让步,通常允许发生spurious wakeup。这对大多数应用程序几乎没有实际影响，因为应该始终在循环中等待一个条件，测试正在等待的状态。具体实现类可以自由地消除伪唤醒的可能性，但建议应用程序程序员始终假设它们可以发生，因此始终在循环中等待。
三种形式的条件等待(可中断的、不可中断的和定时的)在某些平台上的实现便利性和性能特征上可能有所不同。特别是，可能很难提供这些特性并维护特定的语义，比如顺序保证。此外，中断线程的实际挂起的能力可能并不总是能够在所有平台上实现。因此，实现不需要为所有三种形式的等待定义完全相同的保证或语义，也不需要支持中断线程的实际挂起。
需要一个实现清楚地记录每个等待方法提供的语义和保证，并且当一个实现支持线程暂停中断时，它必须遵守这个接口中定义的中断语义。
由于中断通常意味着取消，并且对中断的检查通常不频繁，因此实现更倾向于响应中断而不是正常的方法返回。即使中断发生在可能已经解除了线程阻塞的另一个操作后，也是如此。实现应该记录这种行为。

4. AbstractOwnableSynchronizer

AbstractOwnableSynchronizer是AbstractQueuedSynchronizer和AbstractQueuedLongSynchronizer的父类，是一个可由线程独占的同步器，为创建锁和相关同步器(伴随所有权的概念)提供基础。AbstractOwnableSynchronizer类本身并不管理或使用这些信息，但是子类和工具可以使用适当维护value来帮助控制和监视访问并提供诊断。

5. AbstractQueuedSynchronizer

AbstractQueuedSynchronizer简称AQS，它是一个非常有用的超类，提供一个框架来实现依赖于先进先出(FIFO)等待队列的阻塞锁和相关同步器(信号量、事件等)，许多同步类实现都依赖于它，如常用的ReentrantLock/Semaphore/CountDownLatch。

6. AbstractQueuedLongSynchronizer

AbstractQueuedLongSynchronizer和AQS的区别在于同步状态state是long而不是int，其余的和AbstractQueuedSynchronizer是一致的。这个类用来构建需要状态是64 bit的同步器。

7. LockSupport

源码：

LockSupport提供“创建锁”和“其他同步类的基本线程阻塞原语”。LockSupport的功能和Thread中的Thread.suspend()和Thread.resume()有点类似，LockSupport中的park()和unpark()的作用分别是阻塞线程和解除阻塞线程。但是park()和unpark()不会遇到Thread.suspend和Thread.resume所可能引发的死锁问题。

8. ReentrantLock

ReentrantLock是独占锁。所谓独占锁，是指只能被独自占领，即同一个时间点只能被一个线程锁获取到的锁。ReentrantLock锁可以是公平的和非公平的。ReentrantLock是”可重入的锁”。

9. ReentrantReadWriteLock

ReentrantReadWriteLock是读写锁接口ReadWriteLock的实现类，它包括子类ReadLock和WriteLock。

10. StampedLock

StampedLock是Java8引入的一种新的锁机制，可以认为它是读写锁的一个改进版本。读写锁虽然分离了读和写的功能，使得读与读之间可以完全并发，但是读和写之间依然是冲突的，读锁会阻塞写锁，它使用的依然是悲观的锁策略。如果有大量的读线程，这有可能引起写线程的饥饿。虽然使用”公平”策略可以一定程度上缓解这个问题，但是”公平”策略是以牺牲系统吞吐量为代价的。而StampedLock则提供了一种乐观的读策略,这种乐观策略的锁非常类似于无锁的操作,使得乐观锁完全不会阻塞写线程。
