Java并发06 AbstractQueuedSynchronizer

发表于 2019-06-08 | 更新于 2019-07-29 | 分类于 Java并发

AbstractQueuedSynchronizer又称AQS，它是JUC包中的核心基础组件，提供一个框架来实现依赖于先进先出(FIFO)等待队列的阻塞锁和相关同步器(信号量、事件等)。该类被设计为大多数依赖于单个原子的int值来表示状态的同步器的基础。
AQS使用一个int类型的域state来表示同步状态，state的表示该对象被获取或释放。当state > 0时表示已经获取了锁，当state == 0时表示释放了锁。子类必须定义更改state的方法。该类中的其他方法执行所有的排队和阻塞机制。子类可以维护其他状态字段，但是只有使用getState()、setState(int)和compareAndSetState(int, int)方法来原子操作state的值才能用来同步。
AQS内置的FIFO同步队列完成获取资源的线程的排队工作。如果当前线程获取锁失败，则AQS会将当前线程以及等待状态等信息构造成一个Node结点，并将其加入等待队列，同时阻塞当前线程。当锁释放时会把结点中的线程唤醒并使其再次尝试获取同步状态。
AbstractQueuedSynchronizer的子类应该定义为非public的内部类，用于实现其封闭类的同步属性。AbstractQueuedSynchronizer类不实现任何同步接口，而是定义了像acquireInterruptibly(int)这样的方法，可以调用这些方法来实现具体的公共方法。
该类同时支持独占模式和共享模式的一种或两种(默认是独占模式)。当以独占模式获取到锁时，其他线程尝试获取的操作无法成功。共享模式下多个线程获取可能(但不一定)成功。在不同模式下等待的线程共享FIFO队列。通常子类的实现只支持其中一种模式，但是这两种模式可以全部实现，例如在ReadWriteLock中。只支持独占模式或只支持共享模式的子类不需要定义不支持模式的方法。
这个类有一个嵌套的AbstractQueuedSynchronizer.ConditionObject类用作Condition的实现，这可以用作支持独占模式的子类的Condition实现，此时子类的方法isHeldExclusively()报告同步器是否被当前线程独占、release(int)方法完全释放该对象、acquire(int)最终使对象恢复到以前的状态。AQS没有方法创建这样的condition，所以如果不能满足这个约束，就不要使用它。当然，AbstractQueuedSynchronizer.ConditionObject的行为依赖于其同步器实现的语义。
该类提供内部队列的检查、检测和监视方法，以及condition对象的类似方法。可以根据需要将它们导出到使用AbstractQueuedSynchronizer的类中，以实现它们的同步机制。
该类的序列化只存储底层原子整数的状态，因此反序列化对象的线程队列是空的。需要序列化的子类需要定义一个readObject方法，该方法在反序列化时将其恢复到已知的初始状态。

5. 等待队列

队列结点类定义：

等待队列是CLH(Craig, Landin, Hagersten)队列的变体。CLH锁通常用于自旋锁。当前线程获取同步状态失败时会封装成一个结点并加入等待队列，同时阻塞当前线程；当前序结点释放时当前结点会被signalled。状态变量status并不控制哪个线程被授予锁。当线程是队列的第一个结点时可能会尝试获取锁，但只是参与竞争，并不保证能成功。竞争失败会重新等待。出队是在队头操作，入队是在队尾。
前序结点需要处理取消问题。
