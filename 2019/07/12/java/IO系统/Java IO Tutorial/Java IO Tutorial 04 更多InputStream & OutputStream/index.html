5. FilterInputStream

FilterInputStream是实现自己的输入流过滤器的基类。基本上它只是覆盖InputStream的所有方法并将所有的方法调用传递给FilterInputStream包装的方法。

FilterInputStream没有任何特殊的行为。它旨在成为自定义子类的基类，但一般直接继承InputStream就行。

6. FilterOutputStream

FilterOutputStream是实现自己的输入流过滤器的基类。基本上它只是覆盖OutputStream的所有方法并将所有的方法调用传递给FilterOutputStream包装的方法。

7. BufferedInputStream

BufferedInputStream类为输入流提供buffering。缓冲可以加快IO的速度。与一次从网络或磁盘中读取一个字节不同，BufferedInputStream一次将更大的块读入内部缓冲区。读取一个字节时，是从其内部缓冲区中读。当缓冲区完全读取时，BufferedInputStream再将另一个较大的数据块读入缓冲区。这通常比从InputStream一次读取单个字节快得多，特别是对于磁盘访问和大数据量而言。

可以使用不同的缓冲区大小进行一些实验，以找出哪个缓冲区大小可以在具体硬件上提供最佳性能。最佳缓冲区大小可能取决于使用的BufferedInputStream是磁盘还是网络InputStream。对于磁盘和网络流，最佳缓冲区大小也可能取决于计算机中的具体硬件。如果硬盘一次至少读取4KB，那么使用少于4KB的缓冲区是不合适的。然后最好使用4KB倍数的缓冲区大小。例如，使用6KB也是不太合适的。
即使磁盘一次读取如4KB的块，使用大于此的缓冲区仍然是个不错的选择。磁盘擅长顺序读取数据，这意味着它擅长读取位于彼此之后的多个块。因此，使用16KB缓冲区或64KB的缓冲区(甚至更大)的BufferedInputStream仍然可以提供比仅使用4KB缓冲区更好的性能。
还要注意，某些硬盘具有几兆字节的读缓存。如果硬盘每次从文件读取固定大小(例如64KB)到其内部缓存中，也可以一次将所有这些数据放入BufferedInputStream，而不是多次读操作。多次读取操作将会变慢，并且可能会在读取操作之间擦除硬盘的读缓存，从而导致硬盘将该块重新读取到缓存中。
要找到最佳BufferedInputStream缓冲区大小，需找出硬盘读入的块大小，以及可能的缓存大小，并使缓冲区为该大小的倍数。

B. serialVersionUID

除了实现Serializable接口之外，用于序列化的类还应包含private static final long serialVersionUID。

Java对象序列化API使用serialVersionUIDJava域来确定将要反序列化的对象是否使用相同版本序列化。如果对类进行了重大更改，则还应更改其serialVersionUID值。Java SDK和许多Java IDE包含生成serialVersionUID所需的工具。
如今许多Java项目使用与Java序列化机制不同的机制来序列化Java对象。例如，Java对象被序列化为JSON、BSON或其他更优化的二进制格式。这具有以下优点：对象也可由非Java应用程序读取。例如，在Web浏览器中运行的JavaScript可以本地序列化和反序列化JSON中的对象。顺便说一下， 这些其他对象序列化机制通常不需要Java类实现Serializable。他们通常使用Java Reflection来检查类，因此实现Serializable接口将是多余的 - 它不会添加任何有用的信息。

可以参见https://www.oracle.com/technetwork/articles/java/javaserial-1536170.html。
